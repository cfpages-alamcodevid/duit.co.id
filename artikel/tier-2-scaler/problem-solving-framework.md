Artikel ini buat kamu yang pusing cari akar masalah tiap kali ada problem di bisnis.

Kalau bisnis Anda cuma punya 3 karyawan dan masalahnya cuma "stok kosong" atau "customer marah", belum butuh framework canggih. Selesaikan saja langsung.

Tapi kalau sudah punya 30+ karyawan, masalah kompleks muncul di mana-mana, dan "pusing" cari akar masalahnya, baca terus.

Saya perhatikan banyak pemilik bisnis di Indonesia kalau kena masalah: "Panas dulu, solusi kemudian."

Padahal masalah bisnis itu ibarat es batu.

Lihatnya cuma di atas (es cetar), tapi akarnya di bawah (masalah sistemik).

Saya coba sendiri: Dulu penjualan turun 20%.

Reaksi pertama: "Mana karyawan yang malas? Pecat!"

Ternyata? Masalahnya di sistem stok yang sering kosong, pelanggan ndak dapet barang, alih ke kompetitor.

## Masalah Utama: "Firefighting" (Pemadam Kebakaran)

Bayangkan restoran Anda.

Tiap hari ada masalah:
- "Budi, meja kotor!"
- "Ani, pesanan telat 30 menit!"
- "Bambang, dapur berantakan!"

Anda lari kesana-kemari "memadamkan api".

Tapi ndak pernah tanya: "Kenapa bisa berantakan tiap hari?"

Saya hitung: Bisnis yang cuma "firefight" (pemadam kebakaran) rata-rata buang 30-40% waktu kerja cuma buat selesaikan masalah yang berulang.

Itu bukan bisnis. Itu "pemadam kebakaran" tiap hari.

## Framework 1: 5 Whys (Tanya "Kenapa" 5 Kali)

Ini tool paling simpel tapi paling ampuh.

**Contoh Kasus: Penjualan Turun 20%**

**Why 1:** Kenapa penjualan turun?
→ Karena pelanggan komplain sering telat kirim.

**Why 2:** Kenapa telat kirim?
→ Karena stok di gudang sering kosong.

**Why 3:** Kenapa stok kosong?
→ Karena telat pesan ke supplier.

**Why 4:** Kenapa telat pesan?
→ Karena staf gudang ndak tahu kalau stok sudah menipis.

**Why 5:** Kenapa staf ndak tahu?
→ **ROOT CAUSE:** Karena tidak ada sistem alert stok menipis (masih pakai kertas manual).

Sekarang baru ketemu akarnya: **Ndak ada sistem alert stok.**

Bukan "staf malas" atau "supplier jahat".

Saya coba sendiri: Dari 10 masalah bisnis, 8 di antaranya akar masalahnya di **sistem**, bukan **manusia**.

## Framework 2: Fishbone Diagram (Ishikawa)

Buat yang suka visual, pakai diagram fishbone.

**Struktur:**

```
               [Masalah: Penjualan Turun]
                          |
        ________________|_________________
       |        |        |        |        |
    People    Process   Technology  Environment
       |        |        |        |
    -Gaptek  -Lama    -Sistem    -Kompetitor
    -Malas   -Ribet   -Error     -Ekonomi
    -Ndak    -Salah    -Crash     -Musim
     tahu    input
```

Dari gambar di atas, kita bisa lihat masalah dari 4 sudut:
- **People** (Manusia): Gaptek, malas, ndak tahu SOP.
- **Process** (Proses): Lama, ribet, salah input.
- **Technology** (Teknologi): Sistem error, cras sering.
- **Environment** (Lingkungan): Kompetitor dekat, ekonomi lesu.

Saya sarankan: Pilih 2-3 kategori yang paling "berat" buat diperbaiki dulu.

## Framework 3: Pareto Principle (80/20 Rule)

80% masalah disebabkan oleh 20% penyebab.

**Contoh:**

Dari 100 order yang bermasalah:
- 80 order bermasalah karena **stok kosong** (penyebab A).
- 15 order karena **kurir telat** (penyebab B).
- 5 order karena **salah alamat** (penyebab C).

Fokus perbaiki **penyebab A** dulu.

Saya hitung: Selesaikan penyebab A → 80% masalah beres.

Jangan fokus ke penyebab C yang cuma 5% masalah.

## Langkah 4: Buat Decision Tree (Pohon Keputusan)

Kalau sudah tahu akar masalah, buat "pohon keputusan":

**Kasus: Stok Sering Kosong**

```
[Akar: Ndak ada alert stok menipis]
        |
    [Solusi?]
    /       \
[Manual]    [Otomatis]
 /              \
Cek tiap hari   Pakai software (Majoo/Stokita)
|                |
Makan waktu      Biaya Rp 200rb/bulan
30 menit/hari
|
Masih bisa telat
|
Ganti ke Otomatis
```

Saya coba: Dari decision tree ini, jelas kalau solusi terbaik adalah **software otomatis** (bukan cek manual tiap hari).

## Langkah 5: Action Plan (Rencana Aksi)

Jangan sampe di "solusi" aja.

Buat **Action Plan** yang kongkrit:

**Contoh:**

| Langkah | Aksi | Siapa | Deadline | Biaya |
|---------|------|--------|----------|-------|
| 1 | Cari software inventori | Budi (Admin) | 3 hari | Rp 0 (cari info) |
| 2 | Demo software (3 opsi) | Ani (Gudang) | 1 minggu | Rp 0 |
| 3 | Beli & Setup | Budi + Ani | 2 minggu | Rp 200rb/bulan |
| 4 | Training staf | Ani | 3 hari | Rp 0 (internal) |
| 5 | Go-live | Semua | Tanggal 1 besok | Rp 0 |

Saya sarankan: Tiap langkah harus ada **DONE criteria** (kriteria selesai).

Contoh: "Demo software selesai" → Harus sudah presentasi + punya 3 penawaran harga.

## Metrik Sukses Problem Solving

Jangan selesaikan masalah, tapi minggu depan muncul lagi.

Ukur metrik ini:

**1. Recurrence Rate (Kembali Lagi)**
- Masalah X terjadi 10x bulan lalu.
- Bulan ini? Cuma 2x.
- Berarti recurrence rate turun 80%. Sukses!

**2. Time-to-Resolution (Waktu Selesai)**
- Dulu: Masalah X butuh 3 hari selesai.
- Sekarang: 3 jam selesai (karena sistem alert sudah ada).

**3. Cost Impact (Dampak Biaya)**
- Dulu: Rugi Rp 5 juta/bulan gara-gara masalah X.
- Sekarang: Rugi Rp 500rb/bulan (turun 90%).

Saya hitung: Dengan framework yang benar, efisiensi naik 40-60%.

Masalah jarang muncul, kalau muncul cepat selesai.

## Contoh Nyata: Masalah "Customer Marah"

Saya kasih contoh lengkap:

**Masalah:** Customer marah gara-gara pesanan telat 3 hari.

**5 Whys:**
1. Kenapa telat? → Kurir bilang alamat ndak jelas.
2. Kenapa alamat ndak jelas? → Admin salah input di sistem.
3. Kenapa salah input? → Formulir pemesanan ribet, banyak field yang bingung.
4. Kenapa formulir ribet? → Dibikin 3 tahun lalu, ndak pernah diupdate.
5. **ROOT CAUSE:** Formulir pemesanan ndak user-friendly, butuh revamp total.

**Decision Tree:**
- Opsi A: Revamp formulir (Rp 2 juta, 1 minggu) → Pilih ini!
- Opsi B: Training admin (Rp 500rb, 2 hari) → Cuma nambal, ndak selesai akar.

**Action Plan:**
1. Desain ulang formulir (Budi + Ani) - 3 hari.
2. Testing ke 10 customer - 2 hari.
3. Go-live formulir baru - Tanggal 1.

**Hasil:** 3 bulan kemudian, komplain "alamat salah" turun 95%.

## Budget Problem Solving

Berapa yang harus disiapkan?

| Item | Biaya |
|------|-------|
| Konsultan (Sekali) | Rp 5-15 juta (kalau butuh ahli) |
| Tools (Whiteboard, Sticky Notes) | Rp 200rb |
| Software (Miro, Lucidchart) | Rp 150-300rb/bulan |
| Training Team | Rp 1-3 juta |
| **Total Tahun 1** | **Rp 6,5 - 21 juta** |

Mahal?

Coba bandingkan dengan masalah yang "berulang":

Rugi Rp 10 juta/bulan × 12 bulan = Rp 120 juta/tahun.

Jauh lebih mahal daripada investasi problem solving tools.

## Kesalahan Fatal Problem Solving

Saya lihat banyak yang melakukan ini:

**1. Langsung "tuduh" manusianya**
"Pasti Budi yang malas!" → Ternyata sistem yang error.

**2. Tidak cari Root Cause (Akar)**
Selesaikan gejala aja (kasih bonus supaya karyawan tambah semangat), tapi sistem ndak diperbaiki.

**3. Tidak ada Action Plan kongkrit**
"Oke, kita perbaiki." → Tapi ndak jelas siapa, kapan, gimana.

**4. Tidak ukur hasilnya**
Masalah selesai? Tau dari mana kalau ndak diukur?
